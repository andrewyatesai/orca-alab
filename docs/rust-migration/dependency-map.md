## Notes

Grounding: versions confirmed in /Users/ayates/orc/package.json; the decisive source is /Users/ayates/orc/docs/rust-migration/architecture.md, which sets the strategy this map follows. Existing Rust workspace: /Users/ayates/orc/rust (workspace Cargo.toml + crates/orca-core, pure zero-dep logic already ported and tested).

Cross-cutting strategy (from architecture.md): (1) UI is rebuilt NATIVELY in Rust (orca-ui) — NOT kept in a webview. This makes react/react-dom, monaco, tiptap/prosemirror, the radix/shadcn/lucide/sonner/cmdk/dnd-kit/tailwind component stack, katex, mermaid, and pdfjs the single largest, highest-risk cluster of work, and the Rust GUI-framework choice (GPUI-style custom vs Iced/Slint/egui) is the one open architectural decision gating UI phases. (2) macOS gets a thin Swift/SwiftUI shell owning only native-shell concerns (window/menus/dock/notifications/Sparkle auto-update/code-signing/permissions) over a C-ABI/UniFFI bridge; Linux/Windows use a winit shell. This dissolves most Electron APIs: BrowserWindow→Swift+winit, ipcMain/ipcRenderer/contextBridge→in-process Rust + the bridge, and the desktop-integration APIs map to small crates (arboard/rfd/open/notify-rust/keyring/directories) or the Swift shell. (3) Deps are cargo-vendored under rust/vendor with .cargo/config.toml for offline builds, each stripped via default-features=false + minimal features, and forked-and-trimmed only where a crate is vendored directly. (4) Native C/C++ that can't be rewritten — sqlite and onnxruntime(sherpa) — is vendored as source and built via cc/bindgen behind a safe wrapper (rusqlite bundled; sherpa FFI).

Dedup notes: ssh2 collapses ~10 subsystems into one orca-ssh (russh family) — highest single-crate risk due to agent/parseKey/SFTP parity. node-pty→portable-pty(orca-pty), @xterm/*→alacritty_terminal(orca-terminal, already named in the doc), ws→tokio-tungstenite(orca-relay), tweetnacl→crypto_box (must be byte-for-byte wire-compatible with the existing TS E2EE — write interop tests). reqwest(orca-net) is the shared HTTP core behind electron net/session, fetch, posthog, Linear, and all git-provider REST/GraphQL clients. tokio/std::process(orca-process) is the shared spawn core behind every shelled binary; the load-bearing complexity is the Windows arg-quoting + WSL command-encoding + kill-tree logic, not the spawn itself.

What deliberately stays as external binaries (not rewritten): git, gh, glab (keep CLIs for auth UX; call provider APIs directly for data), agent-browser (vendored native binary, driven over its WS/HTTP), wsl.exe/powershell.exe (Windows execution wrappers), scutil/pmset/defaults (macOS specifics), and curl/Invoke-WebRequest inside generated remote hook scripts (they execute on arbitrary remote hosts, so they cannot be replaced by an in-process Rust client). ripgrep is the exception worth in-processing: embed its own grep/ignore library crates instead of shelling rg.

Biggest risks, in order: (1) the native UI rewrite cluster (react/monaco/tiptap + mermaid/katex/pdfjs which have no clean Rust equivalents — mermaid/katex likely need a vendored-JS-in-rquickjs bridge); (2) ssh2→russh parity; (3) sherpa-onnx/onnxruntime native FFI + model assets; (4) Electron CDP/browser-automation and computer-use input/capture. Lowest risk and best leverage to do first (matches the doc's leaf-first phasing): @parcel/watcher→notify, sqlite→rusqlite, ws→tokio-tungstenite, node-pty→portable-pty, zod→serde, qrcode/yaml/tar, and the orca-net/orca-process cores.
