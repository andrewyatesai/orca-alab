# Parity harness — fresh Rust ports vs. the TypeScript reference

Orca is being **re-implemented fresh** in Rust, using the existing TypeScript as
the reference spec (not a byte-for-byte transliteration). This harness answers
"is the Rust actually doing what Orca does?" by running both implementations
over **one shared input corpus** and diffing their outputs.

It is a **divergence report**, not a strict gate: where Rust and TS agree, you
have real evidence of behavioural parity; where they differ, you get the exact
input + both outputs to decide whether the difference is an intended improvement
or a bug. Intended differences are marked `allowDivergence` and reported without
failing.

## Coverage

71 of the ~88 ported logic modules have adapters — **938 vectors, 936 golden
checks green** via the Rust leg (`cargo run -p orca-parity`). The remaining 17
are out of differential scope (io-edge functions driven only by injected
fs/exec/socket closures, `src/main`-sourced modules, or pure helpers TS keeps
non-exported); see the skip list in `docs/rust-migration/architecture.md`.

## Why this is real evidence (not "trust the port author")

Both compared outputs are **computed live**:

- the **Rust** output comes from running the actual ported crate (`orca-parity`);
- the **TS** output comes from importing and running the real `src/shared/*.ts`.

Neither is hand-authored, so an agreement can't be faked by a mistaken test. The
vectors also carry an `expected` golden (transcribed from each module's
`.test.ts`); the Rust harness checks `Rust == expected` on its own, and the
vitest driver additionally checks `TS == Rust` (and `TS == expected`).

## Layout

```
tools/parity/
  vectors/<module>.json     shared corpus: {function, input, expected?, allowDivergence?}
  dispatch/<module>.ts      TS adapter: vector fn-name + input -> real src/shared call
  dispatch/index.ts         module-name -> TS dispatcher registry
  compare.ts                semantic JSON equality (numbers by value, key-order-insensitive)
  parity.test.ts            vitest driver: TS-vs-Rust diff + golden checks
  rust_outputs.json         generated by the Rust harness (gitignored)
rust/crates/orca-parity/    Rust harness bin: runs vectors through the ports, self-checks goldens
  src/modules/<module>.rs   Rust adapter: vector fn-name + input -> real crate call -> JSON
```

## Run it

```bash
# 1. Rust half — runs the ports, self-checks goldens, writes rust_outputs.json.
#    (Runnable with only the Rust toolchain; no Node required for this leg.)
cd rust
cargo run -p orca-parity -- ../tools/parity/vectors ../tools/parity/rust_outputs.json

# 2. TS half — diffs the live TypeScript reference against rust_outputs.json.
#    (Needs Node + the repo's deps installed.)
cd ..
vitest run --config config/vitest.config.ts tools/parity/parity.test.ts
```

Exit codes: the Rust harness fails if any golden mismatches, a dispatch is
missing, or a vector file is unreadable. The vitest run fails on any unexplained
TS↔Rust divergence.

## Adding a module

1. `vectors/<module>.json` — `{ "module": "<name>", "cases": [ { "function", "note", "input", "expected" } ] }`.
   Seed cases from the module's `.test.ts` (input + expected), then add edge cases.
2. `dispatch/<module>.ts` — `switch (fn)` mapping each vector function to the real `src/shared` export.
3. `rust/crates/orca-parity/src/modules/<module>.rs` — `dispatch(fn, input) -> serde_json::Value`,
   shaping output to match `JSON.stringify` of the TS return (omit absent optionals).
4. Register: add the import/entry in `dispatch/index.ts` and the `pub mod` + match arm in
   `rust/crates/orca-parity/src/modules/mod.rs`.

Keep output shapes identical across the two adapters — that mapping is the only
bespoke code per module; the corpus and comparison are uniform.
