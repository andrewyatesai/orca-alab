# Autoformalization: TypeScript → Trusted Rust (Goal A, the method)

> Goal A is **separate from Goal B**. Goal B (`tools/aterm-vs-xterm/GOAL-B-HANDOFF.md`)
> is hand-written aterm beating xterm. Goal A — this doc — is the **mechanical
> producer** that turns the actively-developed TypeScript backend into its
> Trusted-Rust production form, so the `orca-*` crates stop being hand-maintained
> shadows. aterm is **not** generated by this pipeline.

## The goal

Keep developing orca in TypeScript (fast, ergonomic, full ecosystem) and
**mechanically produce** the Trusted-Rust version of the backend. The Rust stops
being a hand-ported shadow that drifts from live TS and becomes a **derived,
continuously-verified build artifact**. Target: the ~14 dormant `orca-*` crates
(`net`, `ssh`, `relay`, `store`, `session`, `runtime`, `agents`, `text`, `config`,
`crypto`, …). This kills the double-maintenance tax that makes a 73-crate,
bus-factor-one repo unsustainable.

## The honest claim (what "autoformalization" means here — and what it does not)

A generated crate is, precisely:

- **mechanically translated** — produced from the TS, not hand-written;
- **differentially verified** — its output matches the TS on the `orca-parity`
  corpus and on property-fuzzed inputs;
- **proof-carrying safe** — it carries and discharges Trust obligations
  (panic / overflow / UB-freedom via `tcargo`; refinement where a contract is
  derivable), exactly like `orca-git`'s 10 discharged SMT obligations today.

It does **not** claim *formal semantic equivalence to the TypeScript*. TS has no
mechanized formal semantics, so "equivalent to the TS" is established **empirically**
(differential + property testing at scale), not proven. State that plainly in any
write-up. This is autoformalization in the practical sense — lifting loosely-typed
source into a typed, proof-carrying target with machine-checked *safety* and
machine-checked *behavioral parity* — **not** a verified compiler. Over-claiming
here is the same trap as "100% Trust"; don't repeat it.

## What already exists (reuse — do not rebuild)

- **`rust/crates/orca-parity`** — the differential TS-vs-Rust vector harness: the
  equivalence oracle the whole method leans on.
- **`ay` / `ty` / `trustc`** — the prover that discharges safety/refinement
  obligations (present at `~/.cargo/bin/ay`; full toolchain at `~/trust`).
- **`native/orca-node`** (napi) — the integration seam; 3 crates already ship live
  (`orca-core`, `orca-git`, `orca-terminal`).
- **`docs/rust-migration/migration-plan.md`** — the domain→crate decomposition and
  the key seed idea: "TypeScript `.test.ts` files translate verbatim to cargo tests
  for behavioral fidelity."
- **`tools/terminal-bench/gauntlet.mjs`** — the agent-runnable acceptance gate
  (extend it with a `parity` axis; see Stage 4).

## What is new (the method to build)

The pipeline **TS → orca-IR → Rust + contracts**, plus the regen dev-loop.

### Stage 0 — the formalizable TS dialect

Constrain the backend modules to a translatable subset, enforced by an oxlint
ruleset (`config/oxlint-autoformalize.json`) so violations fail the gauntlet, not a
human review:

- explicit types; no `any`/`unknown` escape hatch; no runtime type mutation;
- discriminated unions over class hierarchies; data + pure functions over stateful OO;
- effects (`fs` / `net` / `pty` / `time` / `random` / `git`) only through injected
  **capability interfaces** (already the SSH/relay pattern), which map 1:1 to Rust traits;
- no `eval` / `Proxy` / monkeypatch; deterministic iteration order.

Modules that cannot fit stay TS-only behind the napi boundary. The dialect frontier
is **explicit**, never silent.

### Stage 1 — TS → orca-IR (typed)

Use the TypeScript compiler API (the type checker via `ts-morph`) to lower the
dialect into a small **typed IR**: records, sums, pure functions, capability calls,
total pattern matches. The checker resolves every type, so the IR is fully typed —
**this is the formalization step**: the types become the spec skeleton. Deterministic.

### Stage 2 — orca-IR → Rust

Deterministic lowering for the structured core: records→structs, unions→enums,
functions→functions, capability calls→trait methods, TS `number`→explicit
`i64`/`f64` with **checked** arithmetic. An LLM assists only on residual idioms —
but its output is a **proposal with zero authority**: accepted only if the Stage 4
gates pass. The model adds throughput, never trust.

### Stage 3 — contract synthesis

Attach Trust obligations to the generated Rust:

- **baseline**: panic / overflow / UB-freedom (checked arithmetic + total matches →
  `tcargo` discharges);
- **derived pre/post**: from TS `assert(...)`, invariant comments, and exhaustiveness;
- **refinement**: where the TS states an explicit invariant (e.g. `offset <= len`),
  emit `#[requires]` / `#[ensures]` of the same shape as `orca-git`'s discharged
  obligations.

### Stage 4 — acceptance gates (the oracle)

A generated crate is accepted **iff**, via the gauntlet:

- **PARITY** — `orca-parity` vectors (auto-derived: each TS unit/`.test.ts` becomes a
  vector, plus property-fuzzed inputs synthesized from the IR types) → Rust output ≡
  TS output;
- **SAFETY** — `tcargo` discharges the safety obligations (or `ay` where applicable);
- **BUILD** — compiles offline in the workspace.

Anything red localizes the drift to one module. No human dual-maintenance.

### Stage 5 — integration & promotion

Ship the generated crate behind `native/orca-node` + a flag, **dual-run in shadow**
against live TS in production, and promote it to source-of-truth only after a clean
parity window (the no-data-loss discipline). This is the safe path to "the verified
backend IS the product" — one crate at a time, never a big-bang cutover.

## The development loop (the payoff)

1. A developer changes orca in **TypeScript** (the only source humans edit).
2. `pnpm autoformalize <crate>` regenerates the Rust from the changed TS.
3. `pnpm gauntlet:parity <crate>` runs parity + safety; **green** → that Rust is the
   new production artifact; **red** → the diff shows exactly which module diverged.

Rust becomes a build output. The double-maintenance tax — and most of the merge
friction against upstream — disappears, because divergence is *derived*, not *typed
twice by hand*.

## Bootstrapping (calibrate before you generate)

The 3 already-wired crates (`orca-core`, `orca-git`, `orca-terminal`) are ground-truth
triples — TS, IR-able TS, and the expected hand-written Rust — plus existing parity
vectors. Tune Stages 1–3 until the generator **reproduces** them (output Rust passes
the same parity + safety the hand-written version does). Only then expand, crate by
crate, down `migration-plan.md`'s dependency order: leaf pure-logic first
(`orca-config`, `orca-text`), then `store`/`session`/`runtime`, then `net`/`ssh`/`relay`.

## First milestone (prove the loop end-to-end)

Pick **`orca-config`** (small, pure, IO-isolated). Generate it from its TS via the
pipeline; auto-derive parity vectors from its `.test.ts`; get parity green and
panic-freedom proved; wire it behind the flag. **Success = a one-command
`autoformalize + gauntlet` that reproduces a real crate from TS with parity + safety,
with zero hand-written Rust.** That single green loop is the proof the other 13 crates
are now a generation problem, not a porting problem.

## What this is NOT

- Not a general TS→Rust compiler (scoped to the backend dialect).
- Not proven semantically equivalent to TS (parity is empirical, and says so).
- Not for aterm (hand-written; Goal B).
- Not a reason to fork the TS product surface — it *shrinks* divergence by deriving
  the Rust instead of hand-porting it.
