# aterm UX audit — visual alignment & latency (2026-07-06)

A multi-agent audit of the aterm terminal (Rust/WASM engine integration) answering two questions:
does it look great (fonts, sizes, themes aligned with the design system), and is the end-user
experience free of lag/stutter — with concrete, user-visible improvements.

**Method.** 12 specialized finder agents swept the subsystem (3 visual dimensions, 9 latency/efficiency
dimensions), producing 69 raw findings; a dedup pass merged them to 54 canonical claims; every claim was
then independently re-verified by an adversarial agent instructed to refute it against the real code.
**47 findings were confirmed, 7 refuted.** Findings below include the verifier’s corrections where the
original claim was overstated.

**Snapshot.** All file:line references are against commit `d4e2e6872` (audit-time HEAD, 2026-07-05).
Later commits already overlap some findings — notably `7874fe1e3` and `d6df139b8`
(font-handle registry + lazy classed fallback-font injection), which partially address the cold-open
font-pipeline findings (F-25, F-26, F-28). Re-verify against current HEAD before implementing.

## Verdicts by dimension

### Fonts & typography

The aterm typography foundation is genuinely strong: locale-aware CJK fallback with Han-unification handling, a de-duplicated multi-script fallback chain, emoji and symbol tiers, crisp device-pixel rasterization with triple-trigger DPR reconciliation (matchMedia + ResizeObserver + per-frame reconcile), and live-applying font size/line-height/ligatures. The remaining gaps cluster in three areas, all verified against the real wasm engine: glyph coverage (Nerd Font PUA icons are tofu despite Orca bundling the exact fix for its DOM UI), weight fidelity (variable fonts load at wrong instances, .ttc/bundled families make the Font Weight setting and real SGR bold unreachable on default macOS/Windows configs), and feedback loops (the settings preview ignores family/weight/line-height/ligatures, and family/weight changes silently skip open panes while the UI promises live updates). No letter-spacing control exists but nothing claims otherwise; DPR/anti-aliasing handling matches the design intent and needs no change.

### Theme & color

The core theme pipeline is genuinely strong: the full composed terminal theme (fg/bg/cursor/selection/selection-fg/inactive-selection plus all 16 ANSI colors) is seeded into the engine and live-re-themed in place on open panes across CPU, GPU, and worker paths with scrollback preserved (aterm-controller-theme-mutators.ts, aterm-worker-term.ts:270-291); light/dark toggles repaint immediately via the settings/systemPrefersDark effect (use-terminal-pane-lifecycle.ts:1943-1966); minimum-contrast is handled twice (host-side 4.5:1 floor on the seeded default fg, engine-side per-cell set_minimum_contrast); and adjunct surfaces (IME preedit, link underline, link tooltip, OSC 10/11 replies) all read the live theme or app tokens. The gaps are at the edges rather than the core: search highlights and a few pane-chrome elements use fixed dark-tuned colors that fail on light themes, cursorAccent has no path into the engine despite being an exposed setting, omitted ANSI indices go stale on live re-theme with partial imported themes, and the DEC 2031/DSR 996 scheme signal plus the scrollbar thumb track the app theme rather than the terminal background they actually sit against.

### Chrome & overlays

The aterm overlay chrome is in strong shape at the token level: the link tooltip correctly uses the STYLEGUIDE floating tier (popover surface, hairline border, --radius-sm, the reserved 0-10px-24px shadow, 12px mono for literal URLs), the scrollbar thumb reuses the .scrollbar-sleek color recipe, search-highlight colors intentionally preserve the legacy xterm search look, and the z-index layering (composition 4, scrollbar/IME helpers 5, tooltip 6) is coherent and documented. The gaps are interaction-state and fidelity polish rather than token violations: the scrollbar lacks the hover/drag feedback every other app scrollbar has and never appears for keyboard/search scrolling, the cursor blink free-runs instead of resetting on input, IME preedit renders in the wrong font, and the settings typography preview silently ignores four of the settings it sits next to.

### Input latency (keystroke → PTY)

The local input path is excellent — close to the floor of what an Electron app can do. A keydown encodes synchronously through the wasm engine encoder (deliberately avoiding a worker round-trip, aterm-textarea-input.ts:58-70), flows through the intent guards, is written by the input queue in the same task (with smart coalescing that defeats Chromium's nested-timer clamp), leaves the renderer as a fire-and-forget ipcRenderer.send, and hits node-pty via a direct daemon socket notify with no batching at any layer; the echo direction has interactive bypasses in both the daemon and main-process batchers plus the renderer's presentNow fast path. The four remaining gaps are all at the edges rather than the core: remote-runtime panes eat a needless fixed 8ms input debounce, TUI mouse reports (unlike keys) round-trip through the busy render worker, typing while scrolled up in scrollback never snaps the viewport down (an xterm parity regression that reads as dead keystrokes), and network-backed panes have no local echo so remote typing remains RTT-bound despite all the local echo work having landed.

### Output latency (PTY → pixels)

The byte-to-pixel path is well-engineered for latency: PTY bytes bypass the renderer's output scheduler straight into the engine in arrival order (aterm-output-mirror + writeTerminalOutput up-front mirror), interactive echo bypasses the main-process 8ms batch, the worker parses on message arrival and paints eagerly on the echo nudge, and replies/OSC/bell are pushed rather than poll-drained, with ack-gated backpressure and interactive reserves on the IPC channel. The remaining weaknesses are almost all hot-path amplification rather than added delay: a per-chunk full-buffer search re-run in the worker (the one genuine cliff — it undoes the in-process path's per-frame coalescing), a doubled render per output frame from the eager-present nudge racing the coalesced rAF draw, full-framebuffer putImageData on the CPU paths despite engine-side row damage tracking, and per-cell wasm crossings plus whole-viewport invalidation in the dirty-row snapshot during scrolling. The only fixed pre-parse delay found is the main process's trailing-edge 8ms batch on non-interactive burst heads; everything else on the critical path is within one frame of optimal.

### Frame scheduling

Frame scheduling in aterm is architecturally strong: per-pane draws coalesce onto a single rAF with a hidden-window backstop, all worker panes share one rAF loop, effects follow a genuine idle-to-zero contract (rAF cadence only while animating, a single one-shot timer when settled), keystroke echo has a well-designed eager-present fast path, and app-level hidden panes are suspended on both sides of the worker seam. The gaps are concentrated on the worker seam and edge scheduling: the eager present double-renders every interactive frame because the pending shared-rAF draw is never skipped, the main-thread draw scheduler's consume() contract is not honored by the worker strategy (wedging it after the first frame and silently killing the wake-recovery forced repaint), the in-process fallback keeps painting at near-full rate in a minimized window, and scroll/drag plus cursor-blink phase handling are one polish step behind the echo path.

### Scrolling

The core wheel-to-paint pipeline is in strong shape after the recent perf work: pixel/line/page wheel deltas convert with fractional carry on both the scrollback and TUI paths (aterm-wheel-lines.ts), the wheel handler's redraw lands the same display frame (main rAF fires in the wheel event's frame, posts 'draw', and the worker eager-presents via presentNow — the same fast path the echo fix built), follow-bottom is worker-owned and FIFO-ordered against scroll commands so streaming output cannot yank a scrolled-up reader to the bottom (verified: aterm-worker-terminal.ts followBottomAfter + single command queue), and the scrollbar drag path already predicts display_offset to defeat snapshot lag. The remaining gaps are at the edges rather than the core: mouse-report wheel scrolling still round-trips through the worker (the one input path with the latency profile the keyboard work deliberately eliminated), scrolling is hard-quantized to cell rows, typing from scrollback never snaps to bottom, and the overlay scrollbar lacks keyboard-scroll feedback and track interactivity.

### GPU/CPU renderer strategy

The strategy-selection layer is carefully engineered: a guaranteed CPU fallback at every seam, timeout-guarded GPU init that can't wedge a pane, cached one-time WebGL probes, e2e-only benches behind dynamic imports (zero startup cost), and a well-coalesced shared-worker frame scheduler with an eager interactive present and a smartly skipped overlay clear. The weaknesses are concentrated in three places: policy breadth (Linux Wayland is blanket-denied GPU by an xterm-specific gate, so the default Linux desktop lands on the slowest path), resilience on the now-default worker GPU path (no per-pane context-loss recovery, and crash recovery permanently pins every open pane to main-thread CPU), and the CPU present path throwing away the engine's damage tracking with a full-framebuffer putImageData every frame — including cursor-blink frames on idle panes.

### Render-worker protocol

The render-worker protocol is well engineered at the wire level: fonts ship once per worker generation with transferred buffers, per-pane traffic rides a paneId-stamped envelope with no per-message wrappers, draws coalesce onto one shared rAF loop, STATE snapshots carry dirty-row deltas and omit unchanged selection text, cold reads use an id-correlated query channel with timeouts, and there are no synchronous blocking round-trips anywhere on the interactive path. The remaining overhead is concentrated in work amplification rather than message shape: per-chunk work that should be per-frame (full-buffer search refresh), full renders triggered by commands that change no pixels (hover), per-frame wasm-boundary scans that scale with grid size (dirty rows) or total match count (search rects), a periodic double-serialize clone, and the absence of focused-pane prioritization on the single shared thread. Fixing the first two is low-risk and high-value; the dirty-row and serialize items need small engine-side additions to land fully.

### Pane open & resize

Resize/reflow smoothness is in strong shape: the aterm ResizeObserver commits grid changes at most once per frame with no debounce lag, PTY SIGWINCH is correctly held during divider drags and flushed once on drop, the canvas CSS box is pinned to the rendered backing so undersized frames render crisp instead of upscale-blurred, and DPR changes are covered by a three-tier reconcile (matchMedia tracker, ResizeObserver, per-frame draw-loop guard) with a swapchain-safe present ordering. The weak dimension is cold pane-open latency: the whole boot is lazy and strictly serialized — an unneeded main-thread wasm compile, then a ~200MB OS-font IPC, then worker spawn, then a second wasm compile and full font parse all gate the first prompt paint — and the worker is torn down on last-pane close so a quick reopen re-pays it; custom-font users additionally get a visible typeface swap + reflow after every pane open. The remaining resize-path risk is unbounded engine-rewrap queuing in the single shared worker during divider drags on deep scrollback.

### Search, links & selection

The glue code is thoughtfully engineered for the worker seam — hover hit-testing is rAF-throttled with same-cell guards, the in-process search re-index is coalesced to once per painted frame, selection text is only re-cloned when its range changes, and path-existence probes are LRU-cached. The real gaps cluster in exactly three places: the worker cutover lost the per-frame search-refresh coalescing (full-buffer scan per PTY chunk, paid by every pane sharing the worker) and builds highlight rects by walking all matches with per-match wasm crossings each frame; selection drag has no edge autoscroll at all, hard-capping drag selection at one screenful; and the one-frame snapshot lag races the hover affordances (tooltip never fires when the pointer lands and rests; provider links re-resolve per cell with in-flight cancellation, felt most over SSH). All six findings are surgical fixes to otherwise solid architecture rather than redesigns.

### Main process / daemon PTY

The main-process side of the terminal is in strong shape overall: the renderer delivery layer in ipc/pty.ts is genuinely sophisticated (8ms batching with an interactive echo bypass, ack-bounded per-PTY/global in-flight caps, an active-pane reserve, and hidden-output segregation), the write path is lean fire-and-forget NDJSON, and the Rust daemon's shell-launch layer (client-computed wrappers, daemon-run shell-ready barrier) preserves login-shell env without adding first-prompt latency. The strongest remaining gaps are on the edges of that well-built core: a serial per-spawn scutil resolver probe taxing every new terminal on macOS, the Rust daemon losing the Node daemon's stream batching so bursty output fans out per-read through an expensive per-chunk main-process pipeline, SSH terminals bypassing the batching/backpressure layer entirely, and every daemon-terminal byte being parsed twice locally (daemon engine + main mirror emulator) with checkpoint traffic shipping it a third time through a main-thread JSON.parse.

### Recorded gaps & benchmarks

Recorded-gap hygiene here is unusually good: the design docs are candid (the renderer doc corrects its own overclaims, the worker doc records every accepted staleness/round-trip trade-off), the 2026-07-04 audit was adversarial, and most previously recorded gaps have since been closed (symbol-font tofu, keystroke effects, OSC-aware bell scanning, doc drift, artifact-pin guard). What remains clusters in three places: (1) the benchmark/e2e apparatus validates the non-shipped in-process path — the latency benches honestly measure only the render half on main-thread throwaway engines (PTY round-trip, worker hop, and compositor present all excluded; single-cell payload only), and the end-to-end typing specs have 150-250ms budgets, so nothing anywhere measures the keystroke echo users actually feel on the shipped worker pipeline; (2) the CPU fallback's recorded area-scaled cost (~19ms per single-cell frame at 120x40) is a real ceiling for software-GL users that dirty-rect presentation could largely remove; (3) two smaller recorded accepted-risk items are user-visible (the 1-5s stale shutdown/sleep scrollback capture, and the Scenes placeholder row). The memory bench likewise measures a main-thread CPU wasm instance rather than the shared worker heap production panes live in, but no user-visible harm from that gap could be substantiated, so it is noted here rather than reported as a finding.

### F-01 [high/visual-alignment] Typing while scrolled up never snaps the viewport to the bottom (no scrollOnUserInput equivalent), so keystrokes appear to do nothing

- **Where:** `src/renderer/src/lib/pane-manager/aterm/aterm-textarea-input.ts:178` (finder: scrolling, keystroke-to-pty)
- **Evidence:** The keydown/input/composition senders (aterm-textarea-input.ts:167-181, 216-247, 261-269) pass bytes to inputSink with no scroll handling — the only viewport moves on the keyboard path are Shift+PageUp/PageDown (tryScrollbackPage, 124-139). Follow-bottom only fires when the viewport was already at the bottom before the chunk parsed: worker side `followBottomAfter(wasAtBottom)` gates on `display_offset === 0` (aterm-worker-terminal.ts:124-134), in-process pump likewise (aterm-process-pump.ts:39-54, 'Follow the bottom on new output ONLY if already at the bottom'). Searched the whole terminal-pane layer: scrollToBottom is only invoked by the explicit scrollViewport keybinding (keyboard-handlers.ts:398), visibility-restore follow-output (use-terminal-scroll-visibility-memory.ts:118), and intent enforcement (terminal-scroll-intent.ts:258,292) — nothing on typing, and the PTY input router (pty-connection.ts:2820-2903) has no scroll call either. xterm.js — which the aterm facade replaced wholesale — defaults `scrollOnUserInput: true`, as do Terminal.app/iTerm2/gnome-terminal ('scroll on keystroke'); no mention of dropping it exists in docs/rust-migration/*, so this is a silent behavior regression rather than a documented decision.
- **User impact:** Scroll up to read output, then start typing a command: the viewport stays pinned in scrollback and the shell prompt + echo happen off-screen. The keystrokes appear to do nothing — users read it as a frozen/laggy terminal until they manually scroll back down. A regression versus both the app's previous xterm behavior and every mainstream terminal (xterm.js, iTerm2, kitty).
- **Proposed improvement:** In the textarea input layer (user-initiated input only — not the auto-reply/inputSink seam, which also carries DA/CPR replies): after a successful inputSink() in onKeyDown, onInput, and onCompositionEnd, when the pane is on the normal screen and term.display_offset > 0, call term.scroll_to_bottom(), redraw, and mark followOutput intent. Both methods exist on the worker-backed term shape (aterm-worker-term.ts:272 posts scrollToBottom; display_offset is a sync snapshot read), so the fix is a few lines in the shared textarea-input module and works on both strategies. Gate behind a settings flag mirroring xterm's scrollOnUserInput default-on.
- **Expected gain:** Typed characters become visible immediately when the user was scrolled up — eliminates the 'terminal ignored my keystroke / terminal is frozen' failure mode entirely.
- **Verifier corrections:** Claim is accurate. Two refinements: (a) the keyup path (aterm-textarea-input.ts:209, kitty release events) also feeds inputSink and should be excluded from any snap-on-input to avoid double-firing per keystroke; (b) marking followOutput intent is not optional — without it terminal-scroll-intent.ts:263-267 (pinnedViewport per-write enforcement) would scroll the viewport back to the pin on the next output chunk, undoing the snap. The proposal already mentions marking intent, so this is a confirmation of necessity rather than an error.
- **Verification:** Confirmed end-to-end. (1) No scroll on the typing path: aterm-textarea-input.ts sends bytes via inputSink at lines 178 (keydown), 209 (keyup), 242 (input), 265 (compositionend) with no scroll/redraw/intent call; the only keyboard viewport moves are Shift+PageUp/PageDown (lines 124-139). (2) The facade input() only emits data (aterm-terminal-facade.ts:179-184) and the PTY router routePtyInputData (pty-connection.ts:2820-2903) has replay/presence/intent guards but no scroll. (3) Output-driven follow-bottom is gated on already-at-bottom in both strategies: aterm-worker-terminal.ts:125-126 (`if (wasAtBottom && e.display_offset !== 0) e.scroll_to_bottom()`, wasAtBottom captured at line 132 as display_offset===0) and aterm-process-pump.ts:39-52 — so a scrolled-up viewport never snaps on echo. Worse, per-write intent enforcement actively re-pins the scrolled-up viewport when echo output arrives (terminal-scroll-intent.ts:263-267 scrollToLine on pinnedViewport). (4) Intent tracking listens only to wheel/pointer (terminal-scroll-intent.ts:308-343), and the lifecycle custom key handler only touches intent for PageUp/Home/PageDown/End (use-terminal-pane-lifecycle.ts:1066-1073). (5) scrollToBottom callers renderer-wide are only the explicit keybinding (keyboard-handlers.ts:397-399), visibility restore (use-terminal-scroll-visibility-memory.ts:117-118), and intent enforcement (terminal-scroll-intent.ts:258, 292) — none fire on typing. (6) `scrollOnUserInput` appears nowhere in src, so legacy xterm panes relied on xterm's default true; all panes are now aterm-backed (pane-lifecycle.ts:44 'The pane terminal is an aterm-backed facade'), making this a real parity regression versus the app's prior behavior and mainstream terminals. Severity high: scroll-up-to-read then type-next-command is an everyday flow, and the result (keystrokes+prompt invisible, terminal looks frozen) is exactly the claimed failure mode. Proposed fix locations are sound: term.scroll_to_bottom exists on both term shapes (aterm-worker-term.ts:272; aterm_wasm.d.ts:336) and display_offset is a sync read (aterm-worker-term.ts:176). Minor correction only: the keyup sender (line 209) also sends bytes but should arguably be excluded from the snap to avoid double-triggering; and the fix must also mark followOutput intent, otherwise terminal-scroll-intent's pinnedViewport enforcement (lines 263-267) would fight the snap on the next output chunk — the proposal already notes this.
- **Merged duplicate claims:** Typing while scrolled up in scrollback does not snap the viewport to the bottom (xterm scrollOnUserInput parity gap) — keystrokes look dead

### F-02 [high/visual-polish] Nerd Font / private-use-area icons render as tofu in the terminal even though Orca bundles the exact fallback font that fixes it

- **Where:** `src/renderer/src/lib/pane-manager/aterm/inject-terminal-fallback-fonts.ts:41` (finder: fonts)
- **Evidence:** The engine only receives four fallback tiers: CJK, script chain, emoji, symbol (inject-terminal-fallback-fonts.ts:41-76), sourced from OS system fonts (terminal-fallback-fonts.ts:44-85) — none cover the Nerd Font PUA ranges. The bundled primary jetbrains-mono.ttf covers only U+E0A0-E0A2/U+E0B0-E0B3 (verified by parsing its cmap: 0/16 at U+F000+, 0/12 at U+E5FA+). Loading the actual aterm_wasm engine in Node with the full production fallback chain and rendering U+E5FA (seti git), U+E718 (devicons node), U+F015 (FA home): all three produce pixel-identical output to a guaranteed-unmapped codepoint (.notdef tofu); U+E0B0 renders fine. Meanwhile Orca ships SymbolsNerdFontMono-Regular.woff2 as a CSS @font-face with unicode-range U+E000-F8FF, U+F0000-FFFFD for its DOM UI (main.css:17-24), and the pre-aterm xterm CSS chain deliberately listed 'Orca Nerd Font Symbols' as "bundled PUA fallback for OMP/Powerline glyphs" (layout-serialization.ts:93-111) — that deliberate design was lost in the aterm cutover because the engine rasterizes injected bytes, not CSS stacks.
- **User impact:** Anyone using starship, powerlevel10k, oh-my-posh, or eza/lsd --icons sees tofu boxes for nearly every prompt/file icon in every terminal pane (basic powerline separators survive via the bundled JetBrains Mono, but with the default macOS primary Menlo the seti/devicons/FA/material icon ranges are all missing). The same icons render correctly elsewhere in Orca's UI, making the terminal look broken by comparison.
- **Proposed improvement:** Bundle a TTF/OTF build of Symbols Nerd Font Mono (the OFL license file is already in-repo next to the woff2) as a renderer asset like jetbrains-mono.ttf, and inject it in injectTerminalFallbackFonts via add_fallback_font (or a dedicated PUA tier before the symbol tier), on both the in-process and shared-worker engine builds. Bytes load once per renderer, matching the existing memoized fallback-font pattern.
- **Expected gain:** Powerline/Nerd-Font prompts and icon-mode file listings render real glyphs out of the box — removes the single most visible typography defect for shell-prompt power users.
- **Verifier corrections:** One small clarification, not a refutation: on macOS the default engine primary is not the bundled JetBrains Mono but host-resolved Menlo (default terminalFontFamily 'Menlo', constants.ts:88-99, injected via set_primary_font); modern Menlo also carries the basic Powerline separators, so the observable outcome (separators fine, seti/devicons/FA/material tofu) is as claimed either way. The claim's Node reproduction of pixel-identical .notdef output was not re-run, but it is structurally entailed: no injected face on either engine build maps those codepoints.
- **Verification:** Confirmed by reading every injection path. inject-terminal-fallback-fonts.ts:44-75 injects only cjk/chain/emoji/symbol tiers, sourced from OS script faces at terminal-fallback-fonts.ts:44-159 (PingFang, Geeza Pro, STIX Two Math, Apple Color Emoji, Noto) — none cover Nerd Font PUA icon ranges; the shared-worker build (aterm-worker-engine-build.ts:165-203) seeds the identical set, so both engine builds lack PUA glyphs. The engine rasterizes only injected bytes (aterm_wasm.d.ts:249-251), so the CSS chain listing Nerd fonts (pane-terminal-options.ts:41) is inert for aterm. I independently parsed the bundled jetbrains-mono.ttf cmap (format 12): exactly 7 PUA codepoints (U+E0A0-E0A2, U+E0B0-E0B3); U+E5FA/U+E718/U+F015 unmapped — the claim's coverage numbers are exact. Meanwhile main.css:17-24 ships SymbolsNerdFontMono-Regular.woff2 with unicode-range U+E000-F8FF for DOM UI, and layout-serialization.ts:106 documents the pre-aterm deliberate bundled-PUA-fallback design that the cutover lost — a genuine regression, since under xterm the CSS chain rendered these icons with zero user configuration. Not mitigated elsewhere: applyTerminalPrimaryFont (inject-terminal-primary-font.ts:26-59) resolves only the single user-set family (default 'Menlo' on macOS, constants.ts:99), so icons only work if the user explicitly sets terminalFontFamily to an installed Nerd Font; and the recent 'symbol font tier populated (tofu fix)' covers OS media/technical symbols (terminal-fallback-fonts.ts:71-74), not PUA. Severity high: for the large cohort of starship/p10k/oh-my-posh/eza-icons users of a terminal-centric dev tool, every prompt line and file listing shows tofu constantly, while the same glyphs render fine in Orca's own DOM UI.

### F-03 [medium/visual-alignment] Search highlights are fixed dark-tuned colors; on light themes the active match drops to ~2.9:1 contrast

- **Where:** `src/renderer/src/lib/pane-manager/aterm/aterm-search-overlay.ts:8` (finder: theme)
- **Evidence:** SEARCH_MATCH_FILL 'rgba(92, 74, 0, 0.55)' and SEARCH_ACTIVE_FILL 'rgba(196, 88, 14, 0.6)' are constants (aterm-search-overlay.ts:8-9) painted OVER the finished glyph frame on both the CPU/GPU overlay (aterm-search-overlay.ts:61) and the worker overlay (aterm-worker-overlay.ts:89), with no theme-conditional variant. The justifying comment (lines 4-7) claims parity with 'the xterm search path (TerminalSearch.tsx decorations)', but TerminalSearch.tsx no longer contains any decoration colors (grep for 5c4a00/c4580e finds only the aterm constants), so the parity anchor is stale. Because the fill composites over the glyph, the engine's set_minimum_contrast(4.5) floor (aterm-engine-settings-apply.ts:58) cannot help. On the default light theme (Builtin Tango Light, bg #ffffff, fg #2e3434, terminal-themes/defaults.ts:31-33): active match text composites to rgb(136,74,29) on rgb(220,155,110) ≈ 2.9:1; non-active ≈ 3.8:1.
- **User impact:** In light mode, the very text you searched for becomes the hardest text to read in the pane — the active match renders as a muddy orange band with ~2.9:1 text contrast (below WCAG AA 4.5:1 and below the 4.5 floor the rest of the terminal enforces), and other matches look like dark olive smudges instead of highlights.
- **Proposed improvement:** Derive the two fills from the live theme: keep the current dark-tuned pair when the terminal bg is dark, and switch to light-appropriate highlight tones (e.g. a bright translucent yellow/amber that keeps dark text >= 4.5:1) when isTerminalBackgroundLight() (already exported from terminal-theme.ts) is true. Recompute on updateTheme and forward to the worker overlay alongside the existing highlight rects.
- **Expected gain:** Search matches stay legible and look like highlights on light themes; active match text returns above the 4.5:1 floor the rest of the terminal guarantees.
- **Verifier corrections:** Only trivial nit: non-active match contrast computes to ~3.7:1 rather than the claimed 3.8:1; all other numbers and file:line citations are accurate.
- **Verification:** Verified: fixed dark-tuned fills at aterm-search-overlay.ts:8-9 used unconditionally on both render paths (aterm-search-overlay.ts:61, aterm-worker-overlay.ts:89) with no theme variant anywhere in the aterm glue. The rects paint OVER the blitted glyph frame (comment at aterm-search-overlay.ts:38-40), so the engine-side set_minimum_contrast(4.5) floor (aterm-engine-settings-apply.ts:58; default 4.5 per aterm-engine-settings-apply.test.ts:321) cannot compensate. Parity comment (lines 4-6) points at TerminalSearch.tsx decorations, but grep for 5c4a00/c4580e/decoration colors finds none there — anchor is stale. Default light theme is 'Builtin Tango Light' (terminal-theme.ts:15) with bg #ffffff / fg #2e3434 (terminal-themes/defaults.ts:32-33); recomputed WCAG contrast of the active match is ~2.94:1 (text rgb(136,74,29) on band rgb(220,155,110)) and non-active ~3.7:1 — both below 4.5:1, matching the claim. isTerminalBackgroundLight is exported from terminal-theme.ts (line 177), so the proposed fix is feasible. Severity medium: felt every time a light-theme user searches (common but not constant), and the active match becomes the least-legible text in the pane.

### F-04 [medium/visual-alignment] Theme/setting 'Cursor Text' (cursorAccent) is silently dropped — no path into the aterm engine

- **Where:** `src/renderer/src/lib/pane-manager/aterm/aterm-theme-colors.ts:255` (finder: theme)
- **Evidence:** atermThemeColorsFromITheme (aterm-theme-colors.ts:255-273) reduces the composed ITheme to fg/bg/cursor/selection/selectionFg/selectionInactive/palette — theme.cursorAccent is never read (grep for cursorAccent under pane-manager hits only the ITheme type at terminal-types.ts:154). The engine API has no accent parameter either: set_theme(fg, bg, cursor, selection) (aterm_wasm.d.ts:684) and no cursor-accent setter exists in aterm_wasm.d.ts or aterm_gpu_web.d.ts. Yet both default themes define cursorAccent (terminal-themes/defaults.ts:10, :35), and the settings UI exposes it as an editable color: 'Cursor Text — Color of text under the cursor (block cursor)' (settings/terminal-window-color-groups.tsx:56-68), which composeActiveTerminalTheme merges into the theme (terminal-appearance.ts:156-158) only for it to be dropped at the seed.
- **User impact:** Editing the 'Cursor Text' color in settings does nothing in aterm panes, and imported themes whose cursorAccent differs from the background render the character under the block cursor in the wrong color (the engine's glyph cut-out uses the theme bg, per aterm_wasm.d.ts:466-469).
