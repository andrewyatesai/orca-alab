# Audit — aterm integration + Rust core replacement (2026-07-04)

This audit answers three questions for two migration surfaces:

- **(A)** Orca's integration with the **aterm** terminal engine.
- **(B)** The replacement of Orca's **core logic** with Rust.

…and for each: **what is complete, what is fake or incomplete, and what is next.**

**Method.** A 10-agent fleet mapped six surfaces (migration-plan docs, renderer aterm
integration, main-process daemon cutover, the `orca-daemon` crate, the other 16 Rust
crates, and test/parity/CI infra) and ran four fake/stub/drift hunters. Findings were
adversarially verified (35 verdicts landed: 30 upheld, 5 refuted or materially
corrected). Load-bearing claims were re-checked by hand. Every claim below carries a
`file:line` anchor; anchors were true at audit time — re-verify before editing, code
moves.

**Legend:** ✅ complete · 🟡 incomplete (honest, documented) · 🔴 fake / masking
(fabricates success or hides failure) · 📄 doc drift · 🧵 unwired (real code, no
production consumer).

---

## 0. Correction to the audit's starting premise

The audit began believing `rust/aterm` was an **empty** submodule that blocks the whole
Rust build. **That was a session-start artifact, not the real state.** During the run an
agent ran `build-terminal-addon.mjs`, which auto-initialized the submodule and built
`native/orca-node/orca_node.node`. A verifier then built `orca-daemon` and ran both
parity harnesses green.

As part of this audit the engine was **bumped to the latest release, `v0.18.2607040011`**
(`rust/aterm` pinned to `81538bf2`), up from the prior `fed56825` (~v0.16). The orca
adapter (`orca-terminal`, `orca-daemon`) compiles cleanly against v0.18 and the full
`orca-daemon` test suite passes — the engine bump is behavior-compatible. Artifacts
(native addon + renderer wasm) were regenerated to match the new pin.

What survives from the original "empty submodule" findings is the **fresh-clone risk**,
not a build break: only `build-terminal-addon.mjs` initializes the submodule, there is
**no CI**, and nothing verifies the committed `.wasm`/`.node` match the pin.

## Part A — aterm integration

### ✅ Complete (real and shipping)

- **aterm is the sole terminal engine.** xterm.js is gone from `package.json`; every pane
  is an xterm-API-shaped *facade* over the real wasm engine
  (`src/renderer/src/lib/pane-manager/pane-lifecycle.ts:44-56`), with an explicit "no
  xterm fallback renderer anymore" (`aterm/aterm-pane-open.ts:148`). A missing addon is
  *fatal* on the daemon path, not a fallback
  (`src/main/daemon/headless-emulator-factory.ts:34-46`).
- **The production default is the single shared render worker** — one engine per pane on a
  transferred `OffscreenCanvas`, GPU-in-worker → CPU-in-worker, crash-retire-and-rebuild
  (`aterm/aterm-strategy-select.ts:37-63`). Default flip shipped `16db09b23` (2026-06-29);
  opt-out is e2e-only (`window.__atermWorkerRender=false`).
- **Real through the engine on both paths:** rendering, scrollback
  (`set_scrollback_limit`), search, selection, links, IME, themes + OS color-scheme sync,
  GPU effects (sparkle / cursor-glow), a11y ARIA mirror, native serialize/restore, OSC
  52/7/133 gates, fonts + fallback chain (`aterm/aterm-pane-wiring.ts`,
  `aterm/aterm-terminal-facade.ts`, `aterm/aterm-engine-settings-apply.ts`).
- **Both daemon paths use the same aterm headless engine**
  (`orca-terminal` → `aterm-core`), so renderer and daemon agree on emulation.
- **Rendering-parity tooling is substantive:** xterm-golden conformance gauntlet
  (`tools/conformance`), aterm-vs-xterm fuzzer, snapshot/tui replay parity
  (`tools/terminal-bench`).

### Incomplete / fake

- 📄 **[drift, med] No integrity check ties the committed wasm/napi to the submodule
  pin.** `bump-aterm.mjs:48-55` only *prints* a suggested `git add`; agreement rests on
  operator discipline. `engine()` returns the name `"aterm"` with no version
  (`native/orca-node/src/lib.rs:193`); no CI, no hash check, no runtime handshake.
  → addressed below (drift-check script).
- 🧵 **[unwired, low] The engine's new capabilities are dormant** — `set_symbol_font`
  (symbol-glyph tofu fix) and `note_keystroke` (cursor-comet) exist in glue with zero
  call sites (`aterm/aterm_wasm.d.ts:264,672`). Left deferred: the engine author made
  them opt-in ("the renderer is unaffected until it opts in"), `set_symbol_font` needs
  a new OS-symbol-font source + IPC, and both are cosmetic and can't be visually
  verified headless. Tracked below rather than wired blind.
- 🔴 **[low] "Scenes" settings row is a placeholder for a non-existent feature**
  (`components/settings/TerminalEngineEffectsSection.tsx:220-242`, `control={null}`).
  Honest copy, but discoverable UI surface for an unshipped feature.
- 🟡 **[low] Two bells:** the engine BEL channel is dormant; the live bell is the TS
  byte-scanner in `components/terminal-pane/pty-transport.ts:28`.
- 📄 **[low] Doc + test inversion:** `aterm-single-engine-worker.md` still says "in
  progress" though the default flipped; the general e2e suite forces the **non-default**
  in-process path.
- **[housekeeping]** `pty-transport.ts` / `pty-connection.ts` carry `oxlint-disable
  max-lines` — the exact thing `AGENTS.md` forbids.

### ▶ Next for aterm

1. Add a drift guard tying the two `.wasm` to the submodule pin — **done**
   (`config/scripts/check-aterm-artifact-pin.mjs`, run via `pnpm check:aterm-pin`).
2. Refresh the "in progress" single-engine-worker doc — **done** (now marked shipped).
3. Wire `set_symbol_font` + `note_keystroke` (deferred cosmetic features) with visual
   verification in a running app.
4. Finish the "worker-term refactor" absorbing the monkey-patched worker setters
   (`aterm/aterm-worker-loader.ts:254-304`); promote the worker-on e2e config to default.

## Part B — Rust core replacement

### ✅ Complete (real, wired, shipping)

- **The Rust `orca-daemon` is THE terminal daemon on macOS/Linux** — selection is a
  platform check (`process.platform !== 'win32'`), no env flag, **no Node fallback on
  Unix** (`src/main/daemon/daemon-init.ts:213-222`). Packaged to `Resources/orca-daemon`
  (`config/electron-builder.config.cjs:51-60`).
- **The daemon is genuinely de-stubbed.** All 18 protocol-v18 RPCs dispatch to real logic:
  real PTY spawn, real child exit codes, real `scutil` resolver probe, real PTY-spawn
  health probe, `tcgetpgrp` foreground process, `/proc`-or-`lsof` cwd fallback,
  per-session aterm engine, atomic checkpoint-record `takePendingOutput` with 2 MB bounds,
  real token auth (32-byte urandom, 0600). No canned data (`rust/crates/orca-daemon/src/rpc.rs`).
- **`orca-git` status/numstat/line-count is a real, live cutover** via `orca_node.node`.
  The "43/43 dual-run" claim held up under verification. (Since this audit: the TS
  fallbacks were deleted — the addon is a required dependency and `git-status-stream.ts`
  is Rust-only; the former StatusPorcelainParser is gone.)
- **Both differential parity harnesses are real and green** — pure-logic (`pnpm parity`,
  82 adapters, 1067/1067 goldens) and the live two-legged daemon wire gate
  (`pnpm parity:daemon`, 20 invariants incl. detach/reattach survival).
- **~30k lines of substantively ported, tested crates** exist (orca-core 279 tests,
  orca-git 137, orca-agents 119, orca-config 91, …). No empty shells or `todo!` fakes were
  found in any crate.
- **Warm reattach across app restarts works** (`daemon-init.ts:810-813`; reattach returns
  a real snapshot, `rpc.rs:241-248`).

### Incomplete / fake — the substance of the question

**Genuinely FAKE / masking (fabricates success or hides failure):**

- 🔴 **[high] A dead Rust daemon fails _silently_ to a non-persistent local provider.** On
  Unix, if the binary is missing/times out, the only signal is `console.error(…falling
  back to local…)` (`src/main/index.ts:585`). Terminals then bind to the in-process
  `LocalPtyProvider` (killed on quit — no persistence). The `degraded` flag that
  `pty:management:listSessions` produces has **zero readers in the renderer**, and
  `isDaemonDegraded()` only catches the *preserved-unhealthy* case, not the *never-started*
  case (`src/main/ipc/pty-management.ts:38`). A 60 s fail-open makes it worse
  (`src/main/startup/first-window-startup-services.ts:24,93`). **The user is never told
  their terminals stopped being persistent.** → addressed below.
- 🔴 **[low] `signal` RPC fabricates success** — folds `Some(Err(_))` into `ok`, so a
  bad/unsupported signal name returns success (`rust/crates/orca-daemon/src/rpc.rs:179-185`).
  → fixed.
- 🔴 **[low] The `ORCA_RUST_TERMINAL=0` "node" benchmark arm actually runs aterm**, so any
  node-vs-rust benchmark from `terminal-bench` compares aterm to itself. → doc/comment
  cleanup.
- 🔴 **[med] Lossy UTF-8 decode corrupts data at chunk boundaries** — the daemon decodes
  each 64 KB socket/PTY read with `from_utf8_lossy`, so a multibyte char split across a
  boundary becomes `U+FFFD` on large CJK/emoji pastes and in live stream/checkpoint records
  (the engine grid gets raw bytes and stays correct — so stream and snapshot can
  *disagree*). Acknowledged in-code as a "tracked follow-up"
  (`rust/crates/orca-daemon/src/connection.rs:18-21`, `rpc.rs:386`). → fixed.

**Honestly INCOMPLETE (real gaps, documented, not deceptive):**

- 🟡 **[high] `shellOverride` is silently dropped on Unix.** The adapter forwards it
  (`daemon-pty-adapter.ts:168-187`) but `build_command` reads only cwd/command/env/
  envToDelete and always spawns `$SHELL` (`rust/crates/orca-daemon/src/rpc.rs:309-329`). A
  user who configured a non-default shell for a pane doesn't get it — the highest-impact
  user-visible gap on the current shipping platform. → fixed.
- ✅ **[med] Shell-readiness (OSC-777 marker barrier)** — was "shellState is the constant
  `unsupported`"; now landed on `main` (`92c07a2e9`, "shell-launch completion"). The daemon
  ports `shell-ready-marker-scanner.ts` + `session.ts`'s pre-ready stdin queue in
  `shell_ready_barrier.rs` (11 tests): armed when the adapter sends `shellReadySupported`
  (`daemon-pty-adapter.ts:209,261`), it scans PTY output for the `\x1b]777;orca-shell-ready`
  marker (partial-prefix straddle + held bytes handled), queues stdin until ready, then
  drains; `shellState` reports `pending`→ready, bounded by `shellReadyTimeoutMs`
  (`rpc.rs:360-372`). No longer a gap.
- 🟡 **[high] Windows daemon transport is not migrated.** The Rust daemon's Windows
  `serve()` is still an explicit `Unsupported` stub (`rust/crates/orca-daemon/src/lib.rs`) —
  std offers neither named pipes nor AF_UNIX on Windows without `unsafe` (this crate forbids
  it), so the transport twin lands with its own port; the entire Node daemon stack (node-pty,
  ConPTY) is retained for Windows. Note: the *engine* side de-risked since this audit — the
  aterm submodule pin (`a01f300c`) now carries first-class Windows support in `atpkg`
  (cross-compiles `x86_64-pc-windows-gnu`), so what remains is specifically the daemon
  socket/pipe transport, not the terminal engine. → tracked (highest-risk surface).
- 🧵 **[high, headline] Most of the Rust core is real but _unwired_.** `orca-core`
  (8.8k loc), the `orca-git` **IO tier** (remote/push/pull/rebase/branch ops +
  2,177-loc `source_control_ai`), `orca-config`, `orca-agents`, `orca-runtime` +
  `orca-store`, `orca-ssh`, `orca-relay`, `orca-crypto` — thousands of ported, tested lines
  whose **only consumer is the parity harness**. Production still runs the TypeScript
  equivalents. **Progress since this audit:** the pure `orca-git` *parsers* are now the SOLE
  implementation — the duplicated TS parsers were deleted (main requires the napi addon; the
  SSH relay runs the same Rust core via wasm, `src/relay/git-wasm.ts`): `parseStatusPorcelain`
  + the streaming status parser, `parseNumstat` (`rust-numstat.ts`), `parseWorktreeList`
  (`worktree.ts`), `parseGitHistoryLog` (`rust-git-history-log-parser.ts`),
  `countAdditionsInBuffer`, **plus the first IO-tier slice** — push-target validation now runs
  the `orca_core::git_push_target` value rules in Rust (`validate_git_push_target_rules` →
  `rust-push-target-validation.ts`) while `runner.ts` still executes `check-ref-format`, so
  SSH/WSL/env routing is untouched. This proves
  the **SSH-safe IO-tier decomposition**: Rust owns arg-building + result/error classification;
  TS keeps executing git. It applies to any single-round op (push/pull/fetch/branch ops). What's
  still dead: `computeLineStats` (renderer-only consumer — needs the injectable-param pattern or
  an IPC hop); `NdjsonParser` (per-chunk hot path — napi round-trip likely outweighs the win). The
  **"A bridge" for multi-round IO ops is now BUILT** (`native/orca-node/src/git_executor_bridge.rs`):
  orca-git's sync `GitRunner` logic runs on a libuv worker thread that fires a `ThreadsafeFunction`
  built from a JS git executor and `block_on`s the returned promise — Rust drives (incl. multi-round
  control flow), JS (`runner.ts`) executes git, so SSH/WSL routing is preserved. **First
  multi-round production cutover landed:** `getUpstreamStatus`'s explicit publish-target branch
  (`src/main/git/upstream.ts`) now drives `orca_git::get_upstream_status` over the bridge —
  `check-ref-format` → `rev-parse` → (if found) `rev-list` → (if ahead&behind) `log --cherry-mark`,
  the data-dependent decisions owned by Rust, with the no-upstream swallow + `normalize_git_error_message`
  applied in-process and the result returned as omit-`None` `GitUpstreamStatus` JSON. Required a
  prereq (`has_configured_push_target` added to `orca-core GitUpstreamStatus`) and a message-parity
  fix (the bridge's `GitError.message` now carries git's stderr, matching TS's execFile error). The
  existing `upstream.test.ts` explicit-target cases (exact git-argv sequence, WSL routing, fatal
  normalization) now pass through the Rust path; a 4-dimension adversarial review found 0 confirmed
  defects. **What remains** is the remaining multi-round ops (rebase-continue, retry-with-lease,
  branch cleanup) — and porting `git-configured-branch-target.ts` to Rust before the *effective*-upstream
  (no explicit target) branch can be cut over. → tracked (per-surface, both cutover shapes proven + live).
- 🟡 **[med] Wire-behavior drifts vs the Node daemon:** no synthetic `exit(-1)` on
  write/resize to an unknown session; responses sent for `notify_`-prefixed requests;
  `kill.immediate` and `shutdown.killSessions` ignored; `getSize`/`getCwd` unknown-session
  conventions inverted; `teardownSnapshot` dropped; no stream-data batching; unbounded
  per-client stream channel. → the correctness-affecting ones fixed; batching/backpressure
  tracked.
- 🟡 **[med] Move 1 sub-step 3** (in-daemon `orca-store` persistence) is not built — but the
  reaper and checkpoint/cold-restore **are** done via a *client-side* design
  (`DaemonPtyAdapter` HistoryManager), so the doc's "sub-step 3 not built" is half-stale.
- 🟡 **Move 2** (aterm-gui workspace mode) and **Move 3** (IDE surfaces → native) are not
  started; native macOS is a terminal-only Swift slice, not packaged into a signed `.app`.
- **[dead code]** `production-launcher.ts` and `daemon-pty-provider.ts` have only their own
  tests as importers. → deleted.

**Docs vs code.** The `docs/rust-migration/*.md` snapshots (mostly 2026-06-07) **lag the
code and understate it** — they call aterm an "empty scaffold," describe the cutover as
flag-gated pending work, and show a 1-crate README. Treat the docs as intent, not status.

## Remediation landed in this pass

- **Rust daemon** (`rust/crates/orca-daemon`, `orca-pty`): honor `shellOverride`; a UTF-8
  boundary-carry decoder for the **output pump** (`utf8_stream_decoder.rs`) so PTY output
  split across reads isn't corrupted (the inbound socket-reader half was fixed independently
  on main, commit `e8464f943`); graceful `kill` = SIGHUP + 5 s SIGKILL escalation,
  `immediate` = SIGKILL; `shutdown.killSessions` + socket unlink; synthetic exit(-1) on
  write/resize to an unknown session; `notify_` response suppression; `getSize`
  null-not-throw and `getCwd` throw-on-unknown conventions. `signal` was left as-is —
  verified it already matches the Node daemon (both swallow delivery errors). +12 new tests.
- **Main/renderer**: daemon-degradation now surfaced to the Manage Sessions UI, covering the
  never-started case the old check missed (+test); dead `production-launcher.ts` /
  `daemon-pty-provider.ts` deleted; stale `ORCA_RUST_TERMINAL` comments corrected.
- **orca-git**: parser cutovers — `parseWorktreeList`, `parseGitHistoryLog`, and `parseNumstat`
  now run through the verified Rust parsers behind the napi boundary with the pure TS parsers as
  fallbacks. **First true IO-tier cutover:** push-target validation — the `orca_core::git_push_target`
  value rules run in Rust (`validate_git_push_target_rules`) while `runner.ts` still executes
  `check-ref-format`, establishing the SSH-safe "Rust decides / TS executes" decomposition for the
  rest of the single-round IO tier (push/pull/fetch/branch ops). **The "A bridge"** for *multi-round*
  ops is also built (`git_executor_bridge.rs`): orca-git's sync `GitRunner` logic runs on a libuv
  worker thread that fires a `ThreadsafeFunction` built from a JS executor and `block_on`s the
  returned promise — Rust drives, JS executes git (SSH-safe). Its **first production cutover** is
  `getUpstreamStatus`'s explicit publish-target branch (`get_upstream_status_via_executor`), a
  genuinely multi-round read op. Both cutover shapes are now proven AND live in production.
- **Infra**: `check:aterm-pin` drift guard (`config/scripts/check-aterm-artifact-pin.mjs`)
  asserting the committed wasm matches the submodule pin; stale migration docs refreshed.
  (No GitHub Actions workflow is committed — pushing `.github/workflows/` needs a
  `workflow`-scoped token; the migration gates remain agent-runnable.)

Note: the aterm v0.18 bump and the inbound socket-reader UTF-8 fix were also done
independently on `main` while this work was in flight; this branch adopts main's versions
and keeps only the non-overlapping remediation above.

**Verification:** `cargo test -p orca-daemon` 22 lib + all integration suites green; `pnpm
parity` 1067/1067 goldens + 1152 assertions; worktree napi-parity + suites green; typecheck
+ oxlint clean.
